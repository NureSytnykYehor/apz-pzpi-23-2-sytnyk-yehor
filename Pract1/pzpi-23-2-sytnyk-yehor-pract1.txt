                          3 ОПИС ВИКОНАНОЇ РОБОТИ

      3.1 Проблема умовної логіки

      У багатьох системах об’єкт змінює свою поведінку залежно від поточного
стану. Наївна реалізація такої поведінки зазвичай зводиться до набору умовних
конструкцій:

1   if (state == Draft) { ... }
2   else if (state == Published) { ... }
3   else if (state == Unmoderated) { ... }
4   else if (state == Deleted) { ... }

      Поки станів небагато, такий підхід є прийнятним. Однак із зростанням
системи виникають суттєві проблеми: код починає розростатися, з’являються
довгі ланцюжки умов, а логіка станів змішується з бізнес-логікою. Це ускладнює
розширення та підтримку системи.

      3.2 Ідея патерну State

      Патерн State вирішує цю проблему шляхом інкапсуляції поведінки кожного
стану в окремий об’єкт.
      Основні принципи:
       –   кожен стан представлений окремим класом;
       –   поведінка більше не зосереджена в конструкціях if/switch;
       –   об’єкт-контекст делегує виконання поточному стану;
       –   зміна стану рівнозначна заміні об’єкта стану.

      3.3 Структура патерну

      Патерн State складається з трьох ключових компонентів:
       –   Context – об’єкт, що змінює поведінку залежно від стану. Він зберігає
           посилання на поточний стан і делегує йому виклики, не містячи власної
           умовної логіки.
       –   State – інтерфейс, що визначає контракт поведінки для всіх станів.
                                                                                  5
       –   ConcreteState – конкретна реалізація стану. Реалізує поведінку та може
           керувати переходами між станами, підставляючи новий об’єкт стану у
           Context.
      Мінімальна реалізація патерну виглядає так:

 1   struct State {
 2     virtual void handle(Context& ctx) = 0;
 3   };
 4
 5   struct Context {
 6     State* state;
 7     void request() {
 8       state->handle(*this);
 9     }
10   };

      Механізм    переходів     між   станами   реалізується   через   зміну   стану
безпосередньо в ConcreteState:

 1   struct ConcreteStateA : State {
 2     void handle(Context& ctx) override {
 3       ctx.setState(new ConcreteStateB());
 4     }
 5   };
 6   struct ConcreteStateB : State {
 7     void handle(Context& ctx) override {
 8       ctx.setState(new ConcreteStateA());
 9     }
10   };


      3.4 Переваги та недоліки патерну State

      Серед переваг патерну State виділяють:
       –   відсутність розлогих конструкцій if/switch;
       –   легкість додавання нових станів без зміни існуючого коду;
       –   ізольованість кожного стану – чітка відповідальність.
      До недоліків відносять:
       –   збільшення кількості класів у проєкті;
       –   ускладнення загальної структури системи;
       –   складність відстеження переходів між станами;
       –   надлишковість для простих задач.
                                                                                 6
      3.5 Обмеження патерну State

      Попри всі переваги, патерн State має принципове обмеження: усі переходи
між станами перевіряються виключно під час виконання програми. Це означає
відсутність гарантій коректності на етапі компіляції та можливість некоректних
переходів, які виявляться лише під час виконання.

      3.6 Підхід TypeState

      TypeState є розвитком ідеї патерну State на рівні системи типів. Основна ідея
полягає в тому, щоб закодувати стан об’єкта в його типі:
       –   кожен стан – окремий тип;
       –   операції визначені лише для конкретного стану;
       –   зміна стану означає зміну типу об’єкта;
       –   некоректні переходи виявляються вже на етапі компіляції.
      Приклад реалізації на Rust:

 1   trait State {}
 2   struct Draft;
 3   struct Published;
 4   struct Unmoderated;
 5   struct Deleted;
 6
 7   struct Post<State> {
 8     state: State,
 9   }
10
11   impl Post<Draft> {
12     fn publish(self) -> Post<Published> {
13       Post { state: Published }
14     }
15   }

      У цьому прикладі Post<Draft> і Post<Published> – це різні типи. Метод
publish визначений лише для Post<Draft>, тому його виклик для будь-якого іншого

стану стане помилкою компілятора.

      3.7 Вимоги до мови для TypeState

      Повноцінна реалізація TypeState вимагає від мови програмування наступних
можливостей:
       –   статична типізація – для виконання перевірок на етапі компіляції;
                                                                                   7
      –   параметризовані типи – для кодування стану в типі об’єкта;
      –   move semantics або контроль володіння – для гарантії того, що старий стан
          не буде використаний після переходу;
      –   алгебраїчні типи (бажано) – для спрощення реалізації.
     Підтримка TypeState у різних мовах суттєво відрізняється:
      –   найкраща підтримка: Rust, Haskell, OCaml;
      –   достатня підтримка: Swift, C++;
      –   обмежена підтримка: Python, JavaScript.

     3.8 Порівняння State та TypeState

Таблиця 3.1 – Порівняння підходів
                      State                           TypeState
Рівень перевірки      runtime                         compile-time
Модель                об’єкти станів                  типи як стани
Безпека переходів     часткова                        висока
Складність реалізації середня                         вища

     TypeState підсилює ідею патерну State за рахунок використання системи типів,
переносячи відповідальність за коректність з етапу виконання на етап компіляції.

                                4 ВИСНОВКИ

     У ході роботи було розглянуто патерн проєктування State та підхід TypeState.
     Патерн State дозволяє позбутися умовної логіки шляхом інкапсуляції
поведінки в окремих об’єктах станів. Це робить систему більш гнучкою та
розширюваною, однак збільшує структурну складність. Ключовим обмеженням є
те, що перевірка коректності переходів між станами залишається на рівні виконання
програми.
     TypeState є розвитком цієї ідеї: стан стає частиною типу об’єкта, що
дозволяє перевіряти коректність переходів вже на етапі компіляції та повністю
виключає цілий клас runtime-помилок. Проте повноцінна реалізація TypeState
вимагає відповідної підтримки з боку мови програмування і є більш складною у
застосуванні.
     Таким чином, вибір між State та TypeState залежить від конкретного завдання,
складності системи станів та можливостей мови програмування.

     (це класична схема патерну проєктування **State** / **Стан**):

 3   ```plantuml

 6

12     {abstract} Handle()
13   }

16     Handle()
17   }
18
19   class ConcreteStateB {
20     Handle()
21   }
22
23   Context o-> State : state
24
25   State <|-- ConcreteStateA
26   State <|-- ConcreteStateB
27
28   note bottom of Context
29     state->Handle()
30   end note
31
32   @enduml
33   ```

35   ### Як це працює:
36   * `abstract class State` робить назву класу та його абстрактні методи
     курсивом, як на зображенні.
37   * `Context o-> State` створює зв'язок агрегації (порожній ромб зі
     сторони `Context`) зі стрілкою, що вказує на `State`, та додає підпис
     `state`.
38   * `State <|-- ConcreteState...` створює лінії наслідування з порожніми
     трикутниками.
39   * Блок `note bottom of Context` створює примітку з текстом `state-
     >Handle()` та прив'язує її до класу `Context` пунктирною лінією.
