                                   2 ЗАВДАННЯ

     Практичне заняття присвячене архітектурному аналізу реальної програмної
системи. Завданням є дослідити не лише зовнішній інтерфейс обраної системи, але
й внутрішній устрій: як організовані залежності між компонентами, які архітектурні
рішення визначають масштабованість та надійність.
     Відповідно до завдання необхідно:
      а)   обрати програмну систему для аналізу;
     б)    дослідити її архітектуру;
     в)    підготувати доповідь;
      г)   створити слайди презентації;
     д)    записати відеозапис доповіді тривалістю 10–12 хвилин.
     Для аналізу обрано текстовий редактор Neovim — форк Vim (2014),
написаний мовою C з вбудованим інтерпретатором LuaJIT. Вибір зумовлений
нетривіальною архітектурою: Neovim поєднує асинхронну подієву модель,
міжпроцесовий протокол та вбудовану мову скриптингу у єдину систему.

                       3 ОПИС ВИКОНАНОЇ РОБОТИ

     3.1 Загальні відомості про систему

     Назва системи: Neovim.
     Розробник: Thiago de Arruda та спільнота відкритого програмного
забезпечення.
     Репозиторій: https://github.com/neovim/neovim
     Мова реалізації: C + вбудований LuaJIT.
     Neovim виник у 2014 році як форк Vim з метою усунення структурних
обмежень оригінального редактора: Vim був однопотоковим, тому асинхронний
введення-виведення там був неможливий без переписування ядра, а VimScript мав
суттєві обмеження як мова скриптингу. Рішення полягало у заміні моделі вводу-
виводу на libuv — кросплатформну бібліотеку асинхронних операцій — та у
вбудуванні LuaJIT як основної мови конфігурації та плагінів.
     Сьогодні Neovim — це не лише редактор, а програмована платформа
редагування: через протокол msgpack-RPC будь-який зовнішній процес може
контролювати його або вбудувати як рушій редагування у власне середовище.
За даними Stack Overflow Developer Survey 2025, Neovim п’ятий рік поспіль є
найулюбленішим редактором серед розробників.

     3.2 Архітектурний стиль системи

     Neovim не будувався з нуля — він виник як реакція на конкретні обмеження
Vim. Кожне ключове архітектурне рішення пов’язане з недоліком, який воно усуває.
     Шарова архітектура відповідає на питання стабільності інтерфейсу. П’ять
рівнів — System, Core, C API, Lua stdlib, Client — із суворими однонаправленими
залежностями дозволяють змінювати внутрішню реалізацію ядра, не порушуючи
зовнішні плагіни та GUI. Lua stdlib будується виключно поверх C API і не має
прямого доступу до структур Core.
     Подієво-орієнтована модель є відповіддю на однопотоковість Vim. Усі
джерела подій — клавіатура, LSP-сервери, таймери, файлова система — надходять
до єдиної черги MultiQueue бібліотеки libuv і обробляються асинхронно: жодна
підсистема не блокує введення користувача.
                                                                           5
     Клієнт-серверна архітектура через msgpack-RPC перетворила Neovim з
редактора на платформу редагування. Стабільний двійковий протокол однаково
підходить для зовнішніх GUI (Neovide, nvim-qt), плагінів будь-якою мовою та
вбудовування редактора у IDE як рушія редагування (--embed).

     3.3 Основні компоненти системи




                    Рисунок 3.1 – Шарова архітектура Neovim

     3.3.1 System

     Фундамент архітектури — дві зовнішні бібліотеки. libuv надає єдиний
кросплатформний цикл обробки подій: uv_loop_t обробляє сигнали операційної
системи, uv_timer_t — таймери, uv_fs_event_t — спостерігачів за файловою
системою. LuaJIT — вбудована віртуальна машина Lua з JIT-компіляцією; вона не
запускається як окремий процес, а живе всередині nvim-процесу, що забезпечує
нульові накладні витрати на виклики між C і Lua.

     3.3.2 Core

     Рівень   src/nvim/   містить усю логіку редактора, що не залежить від
інтерфейсу: структуру буфера buf_T з B-деревом рядків memline, управління
                                                                               6
вікнами (window.c) та вкладками (tabpage.c), стан-машину режимів (state.c),
систему автокоманд та undo-redo.

      3.3.3 C API

      Рівень src/nvim/api/ — стабільна публічна межа системи. Він експортує
функції nvim_buf_, nvim_win_ тощо та містить диспетчер msgpack-RPC, що декодує
вхідні повідомлення та маршрутизує їх до відповідних C-функцій.

      3.3.4 Lua stdlib

      Рівень runtime/lua/ реалізує vim.lsp, vim.treesitter, vim.diagnostic,
vim.keymap та інші модулі виключно через vim.api. — публічний C API. Це

принципово: жоден Lua-модуль не звертається до внутрішніх структур Core
напряму.

      3.3.5 Client

      Термінальний інтерфейс користувача (TUI) є єдиним клієнтом, що звертається
до C API напряму. Усі інші — зовнішні GUI, Remote plugins, вбудовані клієнти —
підключаються через msgpack-RPC канали.

      3.4 Взаємодія компонентів

      3.4.1 Ключові структури даних

      buf_T — центральна C-структура буфера:


1   typedef struct {
2     handle_T   b_fnum;        // унікальний ID
3     memline_T b_ml;           // B-tree рядків
4     pos_T      b_last_cursor; // остання позиція
5     // extmarks, highlights, LSP state...
6   } buf_T;


      memline — B-дерево для зберігання рядків буфера. Забезпечує O(log n) доступ

за номером рядка та підтримує своп-файл для великих документів.
                                                                                7
      Extmark — мітка, прив’язана до позиції (рядок:колонка) у тексті. Автоматично

переміщується при редагуванні — це основа для підсвітки від LSP, діагностики і
позначок git.
      Window та Tabpage утворюють ієрархію: одна вкладка містить кілька вікон,

кожне вікно посилається на buf_T, зберігає позицію курсора w_cursor і перший
видимий рядок w_topline. Один буфер може бути відкритий у кількох вікнах
одночасно.
      ModeValue — перелічуваний тип режимів редактора:


1   typedef enum {
2     MODE_NORMAL,
3     MODE_INSERT,
4     MODE_VISUAL,
5     MODE_CMDLINE,
6     MODE_TERMINAL,
7     // ...
8   } ModeValue;


      Поточний режим визначає, як state_enter інтерпретує вхідні події.

      3.4.2 Система каналів msgpack-RPC




                 Рисунок 3.2 – Архітектура каналів msgpack-RPC

      Протокол msgpack-RPC складається з трьох типів повідомлень:
                                                                              8

1   [type, msgid, method, params]      // запит
2   [type, msgid, error, result]       // відповідь
3   [type,        event, args ]        // notification


      Канали розрізняються за транспортом: stdio — Neovim запущений як
підпроцес зовнішнім клієнтом (--embed); socket — для зовнішнього підключення
через TCP або Unix-сокет; pipe — Neovim сам запустив плагін-хост як підпроцес;
pty — для термінальних процесів.

      UI-події — окремий підрежим: зовнішній графічний інтерфейс підписується
через nvim_ui_attach і починає отримувати потік сповіщень win_pos, cursor_goto,
popupmenu_show. Термінальний інтерфейс є внутрішнім клієнтом і звертається до

C API напряму, оминаючи RPC-шар. Зовнішні плагіни запускаються як окремі
процеси — збій плагіна не зупиняє редактор.

      3.4.3 Цикл обробки подій libuv

      Серце Neovim — функція state_enter, що в циклі читає наступну подію від
libuv і передає її поточному обробнику стану. Структура Loop має три черги:

1   typedef struct {
2     uv_loop_t    uv;           // libuv петля
3     MultiQueue events;        // основна черга
4     MultiQueue thread_events; // з фонових потоків
5     MultiQueue fast_events;   // пріоритетні події
6   } Loop;


      Джерела подій: натискання клавіш, LSP-відповіді, таймери, спостерігачі за
файловою системою, RPC-виклики від плагінів. На відміну від Vim, усі ці операції
є асинхронними і не блокують введення.
                                                                            9
      3.4.4 Потік обробки LSP запиту




           Рисунок 3.3 – Потік обробки LSP запиту (на прикладі hover)

      vim.lsp — Lua-шар поверх RPC. Запит vim.lsp.buf.hover() проходить через

buf_request, client:request, channel:rpc_send і асинхронний libuv. LSP-сервер

отримує запит і надсилає відповідь. Зворотний виклик libuv декодує відповідь і
передає обробнику.
      Ключова деталь — vim.schedule: з асинхронного зворотного виклику не
можна напряму викликати nvim API, оскільки це призводить до гонки даних.
vim.schedule ставить виклик у чергу головного циклу:


1   client:request('textDocument/hover', params, function(err, result)
2     vim.schedule(function()
3       vim.lsp.util.open_floating_preview(result.contents)
4     end)
5   end)


      3.4.5 Інтеграція LuaJIT

      LuaJIT вбудований безпосередньо в nvim — не як окремий процес.
Конфігурація — init.lua, плагіни завантажуються через require(). Міст між C і
Lua організований у три шари: Lua stdlib поверх C API поверх LuaJIT VM.
      Простори імен:
                                                                               10

1   vim.api        -- nvim_ C функції
2   vim.fn         -- вбудовані VimScript функції
3   vim.opt        -- опції (:set)
4   vim.lsp        -- LSP клієнт
5   vim.treesitter -- tree-sitter API
6   vim.uv         -- libuv (async I/O)
7   vim.keymap     -- маппінги


      vim.uv надає прямий доступ до libuv з Lua:


1   local timer = vim.uv.new_timer()
2   timer:start(1000, 0, vim.schedule_wrap(
3     function() vim.cmd('write') end
4   ))


      3.4.6 Інтеграція tree-sitter

      tree-sitter — інкрементальний парсер: при зміні буфера повторно аналізується
лише змінений регіон, а не весь файл. Результат — абстрактне синтаксичне
дерево (AST), що використовується для підсвітки через запити, текстових об’єктів,
згортання рядків та обчислення відступів.
      Запити — декларативні S-expression патерни:

1   (function_declaration
2     name: (identifier) @function.name)
3
4   (call_expression
5     function: (identifier) @function.call)


      Граматики компілюються у .so бібліотеки і завантажуються динамічно.
Neovim включає близько 100 граматик за замовчуванням.

      3.4.7 Система плагінів

      Neovim підтримує три типи плагінів.
      Вбудовані Lua-плагіни — виконуються в тому самому процесі, прямий доступ
до vim.api через require().
      Зовнішні плагіни — окремий процес будь-якою мовою: Python (pynvim),
Node.js (neovim npm), Ruby (neovim gem). Спілкуються через msgpack-RPC, збій
плагіна не зупиняє редактор.
      Сумісні з Vim — .vim файли на VimScript для зворотної сумісності.
                                                                                 11
     Важлива архітектурна особливість: уся вбудована функціональність Neovim
— LSP, tree-sitter — реалізована через ті самі публічні програмні інтерфейси, що
доступні плагінам.

     3.5 Масштабованість та надійність

     3.5.1 Режими запуску

     Neovim підтримує три режими запуску без графічного інтерфейсу: headless
(--headless) — для скриптів і безперервної інтеграції; embedded (--embed) —
Neovim керується зовнішнім процесом через stdio; listen (--listen /tmp/nvim.sock)
— Neovim слухає підключення на сокеті.

     3.5.2 Тестування

      –    busted — модульні тести Lua;

      –    функціональні тести через RPC (nvim_command тощо);
      –    screen_tests — перевірка відображення через TestBackend;

      –    безперервна інтеграція на Linux, macOS та Windows.

     3.5.3 Продуктивність

Таблиця 3.1 – Техніки продуктивності Neovim
 Техніка                Застосування
 LuaJIT JIT             конфіг і плагіни у 10–50 разів швидше VimScript
 vim.schedule           безпечний виклик API з асинхронного контексту
 vim.uv                 неблокуючий введення-виведення без співпрограм
 Incremental parsing    tree-sitter повторно аналізує лише змінений регіон AST
 nvim_buf_set_lines     пакетний API мінімізує кількість RPC-звернень


     Уся обробка LSP та tree-sitter виконується асинхронно через libuv, тому
інтерфейс ніколи не блокується на мережевих або дискових операціях.

     У результаті виконання практичного заняття:
      –   досліджено архітектуру текстового редактора Neovim, що поєднує
          шарову, подієво-орієнтовану та клієнт-серверну архітектурні моделі;
      –   визначено, що провідним структурним принципом є шарова архітектура
          з п’ятьма рівнями та суворою ієрархією залежностей;
      –   проаналізовано ключові структури даних: buf_T та memline для зберігання
          тексту, Extmark для анотацій у тексті, Window і Tabpage для ієрархії
          відображення, Loop для циклу обробки подій;
      –   описано    протокол     msgpack-RPC,   що     перетворює   Neovim     на
          програмований сервер редагування зі стабільним публічним протоколом
          для GUI, плагінів та автоматизації;
      –   оцінено архітектурні рішення щодо продуктивності: LuaJIT забезпечує
          10–50-кратне прискорення відносно VimScript, асинхронний введення-
          виведення через libuv не блокує інтерфейс, tree-sitter повторно аналізує
          лише змінені регіони;
      –   сформовано практичні навички архітектурного аналізу програмних
          систем.
     Архітектура Neovim демонструє, як єдина асинхронна основа (libuv),
стабільний публічний протокол (msgpack-RPC) та повноцінна вбудована мова
програмування (LuaJIT) перетворюють текстовий редактор на платформу: GUI,
IDE-фронтенди, тестові фреймворки і CI-скрипти спілкуються з ним через той
самий протокол, що й вбудовані підсистеми.

Event Loop: libuv

main_loop — серце Neovim                                              Джерела подій:

  // Псевдокод `state_enter`
                                                                        → Натискання клавіш (stdin / TUI)
  def state_enter(on_state, data):                                      → LSP відповіді (async через канали)
    do
         event = read_next_event() # Read an event from the OS          → Filesystem watchers ( vim.uv.new_fs_event )
    while on_state(data, event) # Invoke callback for current state
                                                                        → Таймери ( vim.uv.new_timer )
                                                                        → RPC виклики від плагінів
Структура Loop :
                                                                      loop_poll_events перевіряє Loop.uv і

  typedef struct {                                                    Loop.fast_events коли nvim idle, а також при
    uv_loop_t         uv;                                             os_breakcheck .
    MultiQueue       *events;
    MultiQueue       *thread_events;                                   На відміну від Vim, усі LSP/DAP відповіді обробляються
    MultiQueue       *fast_events;                                     асинхронно — вони не блокують введення користувача.
  } Loop;

Tree-sitter інтеграція

Інкрементальний парсинг                               vim.treesitter API:


  Буфер змінився                                       -- Отримати вузол під курсором
